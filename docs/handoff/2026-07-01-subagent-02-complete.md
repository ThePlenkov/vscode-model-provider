## 4. Lessons learned (full extracted rules — see `docs/lessons-learned.md` for the consolidated reference)

The 13 lessons from subagents 00–02 cluster into 7 rules; **the most important are in bold**:

1. **Never use wall-clock sleeps in tests.** Use deterministic stream-splitting at adapter boundaries (or a Transform that splits a known buffer across two macrotasks) — survives slow CI runners. (Discovered twice: PATCHER round 1 of subagent 01, PATCHER round 2 of subagent 02.)
2. **Never reimplement a Node built-in.** Use `Readable.toWeb`/`Writable.toWeb` directly; the SDK's own examples do this. "Do not rewrite" is enforced by all 11 task docs. (Caught in subagent 01 round 1.)
3. **A test that doesn't actually exercise the implementation trivially passes.** Always falsification-trace a new test: ask "what buggy implementation would this still pass?". The `end_of_turn` test took two review rounds before it actually drove notifications through `handleSessionUpdate`. (Subagent 02.)
4. **Public surface must match the contract doc exactly.** Constructor signatures, callback signatures, option field names — if they drift, downstream consumers compile against the doc but fail at runtime. (Subagent 02: `onSessionCreated` callback vs `connectFn` param.)
5. **Try/catch around async lifecycle hooks, and fire the callback AFTER success.** A callback fired before the inner operation completes masks failures. (Subagent 02: `sweepIdle` callback must follow `disconnect()`.)
6. **Update `lastUsed` on every relevant event, not just the start and end.** A 31-min prompt with `idleEvictMs=30min` would be evicted mid-flight if `lastUsed` only updates at start/end. Touch on every chunk. (Subagent 02.)
7. **No-throw callbacks need idempotency and grace.** Cancelling the same entry twice should not double-send `session/cancel`. Rejection paths on RPCs must be caught (`.catch`) to avoid unhandled rejection spam.

Procedural lessons:

8. **Subagent loop: PATCHER → VERIFIER → REVIEWER.** One subagent per round, sequential. Don't run PATCHERs in parallel — rate limits + harder to debug handoffs.
9. **`/act` discipline: substantive reply per thread BEFORE resolve.** Resolve-only is wrong. Each AI reviewer (CodeRabbit, Kilo, Amazon Q, cubic, Gemini, CodeAnt, SonarCloud, CodeScene, Codacy) leaves real findings — address them in the merge commit, not as follow-ups.
10. **CI gate order: build BEFORE typecheck.** `tsc --noEmit` in consumer packages needs `dist/` from producer packages (resolved via the package's `exports` field). The CI workflow already runs build in the typecheck job for this.
11. **`tsconfig.include` references must stay synced with deletions.** Reviewers (e.g., Kilo) flag dead references even when `tsc` itself doesn't fail on them. Always `grep include` after deleting source files.
12. **External check services often require auth the sandbox doesn't have.** Codacy, CodeScene, and similar flag FAILURE/ACTION_REQUIRED without specific details. Document the verdict in the PR, proceed with the merge if GitHub considers it mergeable, address specifics in a follow-up PR once the user grants auth.
13. **In the cloud sandbox, branch switches leave conflicts** when the working tree is uncommitted. Always `git stash` or commit before switching branches. The cloud's `session/` branch tracks compile-time state — use it carefully.

## 5. The subagent loop — operational contract

For each new subagent task:

1. **Create a branch off `main`.** `git fetch origin main && git checkout -b task/NN-task-name origin/main`. Use the same naming convention (`task/03-fs-bridge`, etc.). Note: `base/barebone` from PR #1 was deleted when merged — use main directly.
2. **Dispatch the PATCHER.** Single subagent with a focused `SUBAGENT_CONTEXT_CAPSULE` containing: exact files in scope, exact files forbidden, TDD order, success criteria, reference docs to read first. Reference `docs/agent-tasks/NN-*.md` as the contract; reference `/tmp/agent_bb130001-.../memory/` for context.
3. **Verify locally.** `cd packages/acpify && npx tsc --noEmit -p tsconfig.json && npx vitest run` then `npm run build -ws` then `cd apps/extension && npx @vscode/vsce package --no-dependencies`. Must all be green before REVIEWER.
4. **Dispatch the REVIEWER.** Single subagent with: severity-labelled findings format, contract-compliance checks (settlement rule, FIFO, etc.), codehome lens, minimal-root-cause lens.
5. **Iterate PATCHER on REVIEWER's CRITICAL/IMPORTANT.** Reuse the same task contract. The loop ends when REVIEWER returns CLEAN or MINOR-only.
6. **Commit + push + open PR.** PR title `feat(acpify): <task-name>`. Body must include: summary, what's in the diff, **every review round** with what was fixed, how to verify locally, caveats.
7. **`/act` on the opened PR.** Substantively reply to every AI reviewer thread + resolve. Loop on any IMPORTANT/CRITICAL findings. CodeScene FAILURE + Codacy ACTION_REQUIRED are opaque from sandbox — accept and merge if GitHub says MERGEABLE.
8. **Squash-merge to main.** `--delete-branch=false` (keep the branch until subagent N+1 branches from it — actually it doesn't matter since main is the base). Subject line is the PR title.

The PATCHER does NOT commit — that is your job after REVIEWER returns CLEAN.
